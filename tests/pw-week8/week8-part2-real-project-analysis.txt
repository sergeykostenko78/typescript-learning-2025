TASK 2: READ AND UNDERSTAND AZURE PIPELINES (1 hour)
================================================================================

Your real project has 5 pipeline files:
  - azure-pipelines.yml              (main pipeline)
  - azure-pipelines-basic.yml        (simple version)
  - azure-pipelines-allure.yml       (with Allure reporting)
  - azure-pipelines-saucelabs.yml    (with SauceLabs)
  - azure-pipelines-scheduled.yml    (runs on schedule)

WHAT TO DO:

Start with azure-pipelines-basic.yml (simplest one).

Read it line by line. For each section, write what it does:

  trigger:          ← What triggers this pipeline? (push, PR, schedule?)
  pool:             ← What machine runs the tests? (ubuntu, windows?)
  steps:            ← What steps execute in order?

Typical pipeline steps you'll see:
  - task: NodeTool   ← Installs Node.js
  - script: npm ci   ← Installs dependencies
  - script: npx playwright install  ← Installs browsers
  - script: npx playwright test     ← Runs tests
  - task: PublishTestResults         ← Publishes results to Azure

Then read azure-pipelines-allure.yml — what's DIFFERENT from basic?
Then read azure-pipelines-scheduled.yml — how does the schedule work?

Add to your project-analysis.md:

SECTION 4: CI/CD Pipelines
  - What triggers the basic pipeline?
  - What Node.js version do they use?
  - How do they install Playwright browsers in CI?
  - How does the Allure pipeline differ from basic?
  - What schedule does the scheduled pipeline use? (cron syntax)
  - How do test results get published/visible?

You do NOT need to create your own pipeline this week.
You need to UNDERSTAND these well enough to explain them in an assessment.

DELIVERABLE: Section 4 added to project-analysis.md.

TASK 3: UNDERSTAND ALLURE REPORTING (45 min)
================================================================================

Your real project uses Allure. This is industry-standard test reporting.

WHAT TO DO:

Step 1: Look at the allure-results/ folder
  - What file types are in there? (.json? .xml? .txt?)
  - These are raw results — not human readable yet

Step 2: Check if Allure is in package.json dependencies
  - Look for: allure-playwright, allure-commandline, or similar
  - Note the package names

Step 3: Check playwright.config.ts
  - Find the reporter configuration
  - How is Allure configured as a reporter?

Step 4: If you can run the project locally:
  - Run a few tests
  - Generate the Allure report (usually: npx allure generate allure-results)
  - Open it (usually: npx allure open)
  - Explore: dashboard, suites, graphs, timeline

Add to your project-analysis.md:

SECTION 5: Allure Reporting
  - What packages enable Allure in this project?
  - How is Allure configured in playwright.config.ts?
  - What information does the Allure report show?
  - How does Allure differ from Playwright's built-in HTML reporter?

DELIVERABLE: Section 5 added to project-analysis.md.

TASK 4: ENVIRONMENT CONFIGURATION (30 min)
================================================================================

Your real project has conf/envs/.env.stg (staging environment).

WHAT TO DO:

Step 1: Open .env.stg
  - What variables are defined? (BASE_URL? API_KEY? credentials?)
  - How does the project read these values? (look for process.env in code)

Step 2: Check playwright.config.ts
  - Is there a dotenv import? (import dotenv or require('dotenv'))
  - How does baseURL get set from env variables?

Step 3: Check if there are other env files (.env.prod, .env.dev)
  - How would you switch between environments?

Add to your project-analysis.md:

SECTION 6: Environment Configuration
  - What environment variables does the project use?
  - How are env variables loaded into Playwright config?
  - How would you run tests against a different environment?
